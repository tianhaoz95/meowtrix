Developer Documentation

How the project is put together — start here before changing anything.

Overview

Meowtrix is deliberately small and dependency-light. The frontend is plain ES scripts served directly from public/ — no bundler, no modules, no build step. Files share state through the global scope and are loaded with ordered <script> tags. The backend is a single server.js with two responsibilities.

There is no linter or build step. A Playwright E2E test suite lives in tests/ and can be run locally or inside Docker — see Docker-based testing. After editing frontend code, reload the page (or use ./start.sh for auto-reload).

📐

The canonical architecture notes live in CLAUDE.md at the repo root. This page mirrors them in a browsable form.

Dev commands

npm install        # install deps (node-pty compiles natively)
npm start          # run the server on PORT (default 3000)
./start.sh         # dev mode: nodemon + browser hot-reload (HOTRELOAD=1)
./demo.sh          # serve public/ statically in serverless demo mode
npm run test:e2e   # run Playwright E2E tests against a locally running server
npm run screenshots # capture landing-page showcase screenshots

Docker-based testing

Because node-pty compiles a native addon, binaries built on macOS are incompatible with Linux Docker images. Two workflows exist to work around this.

Full E2E suite via Docker Compose

The repo ships a docker-compose.test.yml that spins up two services:

  • meowtrix — builds the app image from Dockerfile, binds the container port to host port 9234, and exposes a health-check endpoint so Playwright waits until the server is ready.
  • playwright — uses the official mcr.microsoft.com/playwright image, installs dev-only deps (including Playwright itself), and runs npx playwright test with BASE_URL=http://meowtrix:9123.
# Run the full test suite in a clean Linux environment
docker compose -f docker-compose.test.yml up --build --abort-on-container-exit

# Tear down volumes afterwards (optional)
docker compose -f docker-compose.test.yml down -v
⚠️

The node_modules directory inside the container is intentionally isolated from the host via an anonymous volume — this prevents macOS-compiled native binaries from polluting the Linux container.

Interactive dev container via docker-run.sh

For live development against a containerised server (e.g. to test on a different OS or to expose the app on a LAN), use docker-run.sh:

# Build the image and start Meowtrix on a random free host port
./docker-run.sh

# Optionally pass a specific workspace directory to mount
./docker-run.sh /path/to/your/project

The script auto-detects a free host port, mounts the repo as /app inside the container, and binds your chosen workspace directory to /workspace. Settings are persisted to the meowtrix-dev-settings named Docker volume across restarts. Hot-reload (HOTRELOAD=1) is enabled, so editing files in the mounted /app directory triggers a live server restart.

Debugging hints

Server-side (Node.js)

  • Verbose logging: Set DEBUG=1 before starting the server (DEBUG=1 npm start) to enable additional console output for PTY lifecycle events and WebSocket messages where those log branches exist.
  • Inspect mode: Run node --inspect server.js and open chrome://inspect in Chrome to attach the V8 debugger — useful for stepping through WebSocket handlers and REST endpoints.
  • PTY output buffer: Each PTY keeps a rolling ~200 KB ring buffer in ptys.get(id).buf. You can dump it from a Node REPL or a breakpoint to understand what a client would see on reconnect.
  • git API endpoints: Hit them directly with curl to rule out frontend bugs, e.g. curl http://localhost:9123/api/git/status?root=/path/to/repo.

PTY replay issues

If the terminal looks garbled after a refresh, the likely cause is a width mismatch between the PTY's recorded grid size and the current pane size. The server sends pty:created {cols, rows} (the PTY's generation size) before the buffer on reconnect; the client snaps xterm to that size, writes the buffer, then re-fits. Check the onPtyRestore / onReplayDone path in pane.js.

Frontend (browser DevTools)

  • Script load order: All frontend files are globals — if a function is undefined at call-time, the most likely cause is a missing or out-of-order <script> tag in index.html. Check the Network panel to confirm all scripts loaded.
  • WebSocket traffic: Open DevTools → Network → WS, select the socket connection, and watch the Frames tab. Every PTY input, schedule update, and session broadcast is visible there in real time.
  • Monaco editor: Monaco is lazy-loaded from CDN by ensureMonaco(). If the editor tab is blank, check the Console for AMD loader errors and confirm the CDN URLs are reachable from your network.
  • Session/workspace state: The active workspace JSON is written to ~/.meowtrix/session.json on the host. Inspect or delete it to reset the layout without restarting the server.

Single server, two jobs

Everything backend is in server.js:

  • HTTP — serves public/, the settings/session REST API, and the embedding proxy.
  • WebSocket — PTY lifecycle (pty:create/input/resize/destroy) plus the active-session coordination messages.

PTY lifecycle & persistence

PTYs outlive WebSocket connections. The ptys Map keys each PTY by id and keeps a rolling ~200 KB output buffer. On reconnect, a pty:create for an existing id replays that buffer and reattaches the live stream instead of spawning a fresh shell — this is what makes refreshes and network blips non-destructive.

Closing a browser tab or dropping the WebSocket only detaches listeners; the shell keeps running. PTY state is in-memory only, so restarting the server kills every shell.

Single-session coordination

Only one client is the active session; every other connected client (across browsers and devices) sees the inactive overlay. The flow lives in server.js + public/app.js:

  • The newest client to send session:claim wins.
  • Others receive a session:state broadcast and drop to the overlay until they press Move session here.
  • When the active socket drops, the server hands off to the most-recently-connected remaining client.

This is intentionally not done via BroadcastChannel/localStorage — server coordination is what makes it work cross-device, not just across tabs in one browser.

Workspace state

The layout is serialized to the server, not the PTYs. captureWorkspaceState / restoreWorkspaceState (public/app.js) persist the pane/split/tab tree — including flex ratios and PTY ids — to ~/.meowtrix/session.json via /api/session.

Restoring rebuilds the DOM and reconnects each terminal to its still-alive PTY by id. Only the active session may write (an inactive tab holds a stale layout); saves are debounced and a beforeunload beacon flushes the final state.

Embedding proxy

The /proxy route fetches pages server-side, strips X-Frame-Options / CSP frame directives, and rewrites href/src/action URLs to route back through the proxy — letting otherwise un-embeddable sites render in a browser pane.

Two route shapes exist for a reason:

  • /proxy?url= — initial loads.
  • /proxy/<encoded> — target in the path, so GET-form submissions don't wipe the target when the browser replaces the query string with form fields.

Google, sign-in flows, and heavily bot-protected sites still won't embed.

File System & Git APIs

The code editor is powered by client-side logic in public/editor.js communicating with Node-side APIs in server.js:

  • Directory listing & Reads/Writes: Shipped over simple JSON endpoints (/api/fs/list, /api/fs/read, /api/fs/write). Because the user already has full terminal/shell access, file reads and writes run with the same host permissions as the server process.
  • Git Operations: Backed by /api/git/* endpoints in the server that safely shell out using Node's execFile (avoiding the shell injection risks of exec). Endpoints include status (which runs git status -z and parses branch tracking, staged, and unstaged modifications), filediff (which yields HEAD, index, or worktree buffers), and POST actions for stage, unstage, discard, commit, push, and pull.
  • AI Commit Message Generator: If a staged diff exists, the UI contacts the AI commit endpoint (using locally configured environment keys or fallback models) to generate a detailed commit description from the staged git diff.

Schedules Timer

The scheduled Enter key press mechanism runs entirely server-side in server.js (the schedules Map, keyed by PTY ID). Living on the server allows timers to survive temporary browser disconnects or client reloads.

  • Creation: Client triggers timer setup via a WebSocket message schedule:create {ptyId, fireAt}. The server sets a setTimeout for that exact millisecond.
  • Execution: When the timeout fires, the server writes a carriage return (\r) directly to the target PTY stream and deletes the schedule, broadcasting the updated schedules state back to all clients.
  • Stateless Rendering: The frontend public/schedule.js registers for schedule:state broadcasts. It is a pure visual overlay layer that renders countdown clocks and blurs terminal input while a schedule is active, but stores no timer state itself.

Supervisor-Driven Self-Updates

Self-updates are designed to run git-based updates seamlessly while protecting the running terminal processes:

  • Hourly Check: The server runs a background check against upstream git coordinates. When updates are found, it broadcasts update:state to all sockets.
  • Clean Application: A POST request to /api/update/apply triggers a git pull. If dependencies have changed, a local install runs. The server then calls process.exit(0).
  • Supervisor: If installed as a launchd/systemd service (defining MEOWTRIX_SUPERVISED=1), the process manager automatically detects the clean exit and restarts the process using the fresh codebase. If run manually, the server exits and asks the user to restart it.

Frontend module map

All files are global-scope and load order matters. In index.html: demo.js → ws.js → pane.js → editor.js → layout.js → app.js → mobile.js → settings.js → palette.js → schedule.js → update.js → combo.js → pet.js. Cross-file calls are plain function references (e.g. wsSend, getAllPanes, applyStickyMods), not imports.

Serverless demo

Because the frontend is bundler-free static files, it can run with no backend at all. public/demo.js activates when the URL has ?demo and stands in for the server:

  • Terminals → JS REPL. demoBackend speaks the exact same pty:* / session:* message shapes ws.js expects; each "PTY" is an in-browser JavaScript REPL with its own line editor, history, and eval()-based evaluation.
  • REST → localStorage. A fetch shim intercepts /api/settings and /api/session and backs them with localStorage, so app.js and settings.js are untouched.
  • Browser panes load URLs directly (no proxy), so only embeddable sites render — with an on-screen hint to that effect.

The seams are narrow: ws.js branches on window.DEMO_MODE for the socket, and fetch is shimmed for the API. Everything else — pane.js, layout.js, app.js, settings.js, mobile.js — is shared verbatim between the real and demo paths. Run it locally with ./demo.sh.

Distribution

install.sh clones/updates into ~/.meowtrix/app, installs production dependencies, and writes a meowtrix launcher to ~/.local/bin. With --service it installs a launchd agent (macOS) or systemd user service (Linux) that auto-starts on login and restarts on crash. The README's install commands curl this script from GitHub.

Project site & deploy

This site lives in website/ (landing page + these docs) and is deployed to GitHub Pages by .github/workflows/deploy-pages.yml on every push to main.

The workflow assembles a deploy artifact rather than publishing a branch: it copies website/* to the site root and the app's public/* into /demo/, then publishes via actions/deploy-pages. That's why the live demo at /demo/?demo is the same code as the app — it's literally public/ served statically.

🚀

Pages source is set to GitHub Actions (not "deploy from a branch"). All links in the site are relative so it works under the /meowtrix/ project-pages base path.
